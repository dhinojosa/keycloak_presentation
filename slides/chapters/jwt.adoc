=== JSON Web Token

[.column]
--
* Compact token format
* Carries named claims
* Usually signed
* Easy to pass through HTTP
* Format does not define purpose

[.small]
Spec: https://www.rfc-editor.org/rfc/rfc7519[RFC 7519: JSON Web Token]
--

[.column]
--
image::jwt-logo.svg[JSON Web Token logo,350]
--

[.notes]
--
JWT tells us how a token is represented, not what the token is for. Access Token
and ID Token describe jobs. JWT describes a format that either token may use.
--

=== JWT Has Three Parts

[cols="25b,75"]
|===
| Part | Contains

| Header
| Token type, signing algorithm, and key ID

| Payload
| Claims about the token, subject, and context

| Signature
| Cryptographic protection against modification
|===

[source,plaintext]
----
header.payload.signature
----

=== A Decoded JWT

[source,json]
----
Header:  { "alg": "RS256", "kid": "42", "typ": "JWT" }

Payload: { "sub": "248289761001", "exp": 1752188400 }

Signature: <cryptographic bytes>
----

WARNING: Decoding makes the header and payload readable. It does not make their claims trustworthy.

[.notes]
--
Anyone can Base64URL-decode or construct a JWT header and payload. Trust begins
only after signature verification and required claim validation, covered in the
JWKS chapter.
--

=== Claims

* `sub` - subject
* `iss` - issuer
* `aud` - intended audience
* `exp` - expiration
* `scope` / `roles` - authorization inputs

[.small]
More registered claims: https://www.iana.org/assignments/jwt/jwt.xhtml#claims[IANA JWT Claims Registry^]
