== OAuth 2.0
image::iStock-1181695869.jpg[background, size=cover]

=== OAuth 2.0

* Delegates access without sharing passwords
* Issues Access Tokens
* Uses scopes to limit requested access
* Defines authorization

[.small]
Spec: https://www.rfc-editor.org/rfc/rfc6749[RFC 6749: OAuth 2.0]

=== Am I Logging in to OAuth?

* *Question:* I am logging in to OAuth, right?
* *Answer:* No. You are logging in to Keycloak
* Keycloak performs the authentication
* OAuth defines the authorization

[.notes]
--
Keycloak verifies who the user is. OAuth defines how a Web Application receives
permission to access something on the user's behalf. OpenID Connect later gives
the Web Application information about who authenticated.
--

=== Actors in This Flow

[cols="35b,65"]
|===
| Actor | What It Does

| Browser
| Carries redirects and cookies

| Web Application
| Starts login, handles the callback, and stores its session

| Keycloak
| Authenticates the user and issues tokens
|===

=== Access Token

* *Analogy:* An Access Token works like a temporary door badge
* Caller presents it to a Web Application endpoint that requires it
* Can use either format:
** JWT - readable claims protected by a signature
** Opaque - identifier with no claims to inspect
* Keycloak normally issues signed JWT Access Tokens
* May contain identity claims, but its purpose is access

[.notes]
--
Think of an Access Token as a temporary door badge. The badge grants access to
specific doors for a limited time. That does not make it the Web Application's
login record. OpenID Connect supplies the ID Token for that purpose.
--

=== Access Token on the Wire

[source,http]
----
GET /orders HTTP/1.1
Host: orders.example.com
Authorization: Bearer eyJhbGciOiJSUzI1NiIsImtpZCI6IjQyIn0...
----

* Web Application validates the Access Token before handling `/orders`
* Anyone holding a bearer token can use it
* Short expiration limits how long a stolen token works

=== Inside a Keycloak Access Token

[source,json]
----
{
  "iss": "https://identity.example.com/realms/workshop",
  "sub": "8f25c7b1-...",
  "aud": "orders",
  "exp": 1752188400,
  "scope": "orders:read",
  "realm_access": { "roles": ["user"] }
}
----

NOTE: This is representative, not an exact schema. Claims depend on configured scopes, role mappings, and protocol mappers.

=== Reading the Access Token Claims

[cols="30b,70"]
|===
| Term | In the Previous Token

| Claim
| Any named value, such as `iss`, `sub`, `aud`, or `exp`

| Scope
| `orders:read`

| Role
| `user` inside `realm_access.roles`
|===

WARNING: A claim is an authorization input, not permission by itself.

=== Access Token and Refresh Token

[cols="35b,65"]
|===
| Token | Where It Goes

| Access Token
| Endpoint that requires it; intentionally short-lived

| Refresh Token
| Keycloak token endpoint only; longer-lived and protected aggressively
|===

[source,yaml]
----
access_token:  eyJ...
refresh_token: eyJ...  # Treat as opaque
----

NOTE: A Refresh Token is issued only when the flow, requested scopes, and Keycloak configuration allow it.

=== Refresh Token Flow [Animation]

. Keycloak issues an Access Token and, when applicable, a Refresh Token
. Web Application uses the Access Token until it expires or is rejected
. Web Application sends the Refresh Token to Keycloak's token endpoint (`/protocol/openid-connect/token`)
. Keycloak returns a new Access Token and may rotate the Refresh Token
. Rejected or expired refresh requires the user to authenticate again

=== Refresh Token Rotation

* Keycloak may return a replacement Refresh Token
* Web Application stores the new token and discards the old one
* Reusing the old token may indicate theft
* Rejected refresh requires the user to log in again

=== Where Tokens Stay

[cols="28b,36,36"]
|===
| Architecture | Browser Holds | Server Holds

| Server-Side Web Application
| Random session cookie
| Access Token and Refresh Token in session storage

| Browser-Only SPA
| Access Token in JavaScript memory
| Endpoint validates the incoming Access Token
|===

NOTE: A Refresh Token in browser JavaScript requires a deliberate risk decision. A server-side Web Application keeps it off the Browser.

=== Presenting and Storing Tokens

* `Authorization: Bearer` presents an Access Token to an endpoint
* An HTTP header carries the token; it does not store it
* Server-side Web Application: Browser sends a session cookie
* Browser-only SPA: JavaScript sends the Access Token header
* Do not embed Access Tokens or Refresh Tokens in HTML

WARNING: PKCE protects the authorization code exchange. It does not protect stored tokens.

[.notes]
--
Keeping an Access Token in JavaScript memory limits persistence but does not make
it immune to XSS. Browser local storage and session storage are readable by page
JavaScript. A Backend-for-Frontend keeps OAuth tokens server-side and gives the
Browser an HttpOnly session cookie instead.
--

=== Do Not Handle User Passwords

* Browser sends user credentials only to Keycloak
* Web Application redirects the Browser to Keycloak
* Keycloak handles MFA, passkeys, federation, and password policy
* Web Application receives an authorization code, never the password
* Disable Direct Access Grants unless a documented legacy requirement needs them

WARNING: Do not place the Web Application between the user and Keycloak when credentials are entered.
