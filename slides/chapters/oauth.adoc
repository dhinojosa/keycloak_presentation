== OAuth 2.0
image::iStock-1181695869.jpg[background, size=cover]

=== OAuth 2.0

* Authorization framework
* Delegates access without sharing passwords
* Issues Access Tokens
* Uses scopes to limit requested access
* Does not prove who the user is

[.small]
Spec: https://www.rfc-editor.org/rfc/rfc6749[RFC 6749: OAuth 2.0]

=== OAuth Is Not Authentication

* OAuth delegates access to a protected resource
* An Access Token tells an API what access was granted
* OAuth does not define a login result for the client
* Do not use an Access Token as proof of identity

[.notes]
--
The authorization server may authenticate a user during an OAuth flow, but OAuth
does not tell the client who that user is. The Access Token is intended for the
API and represents delegated access. OpenID Connect later adds an identity layer
and gives the client a defined login result.
--

=== Access Token: Job vs Format

* Access Token describes the token's job
* JWT describes a token format
* Keycloak usually issues Access Tokens as signed JWTs
* OAuth also allows opaque Access Tokens

[.notes]
--
An Access Token and a JWT are not competing kinds of token. "Access Token" says
what the token is for; "JWT" says how information is represented and protected.
An opaque Access Token has the same job, but the API cannot validate it as a JWT
and may need token introspection.
--

=== Inside a Keycloak Access Token

[source,json]
----
{
  "iss": "https://identity.example.com/realms/workshop",
  "sub": "8f25c7b1-...",
  "aud": "orders-api",
  "exp": 1752188400,
  "scope": "openid profile orders:read",
  "realm_access": {
    "roles": ["user"]
  },
  "resource_access": {
    "orders-api": { "roles": ["order-reader"] }
  }
}
----

NOTE: This is a representative example, not an exact schema. Claims depend on client scopes, role mappings, and protocol mappers.

[.notes]
--
The API uses `iss`, `aud`, and `exp` when deciding whether to accept the token.
The `sub` identifies the subject. Scope and role claims provide authorization
inputs, but the application still decides what those values permit locally.
--

|===
| Participant | Role

| Resource Owner
| User or system that owns the access

| Client
| Application requesting access

| Authorization Server
| Keycloak

| Resource Server
| Protected API or application that accepts Access Tokens

| Access Token
| Credential presented to the API
|===

=== PKCE

Proof Key for Code Exchange

. Application creates a random secret called the `code_verifier`
. It hashes that secret into a `code_challenge`
. Browser redirects to Keycloak with the `code_challenge`
. Application exchanges the authorization code for tokens and sends the original `code_verifier`
. Keycloak hashes the verifier and checks that it matches the earlier challenge

[.small]
Spec: https://www.rfc-editor.org/rfc/rfc7636[RFC 7636: PKCE]

=== Authorization Code Flow

* Browser-based login
* User authenticates with Keycloak
* Client receives an authorization code
* Client exchanges the code for tokens
* PKCE protects the code exchange

=== Device Authorization Flow

* Used by TVs, game consoles, CLIs, and appliances
* Device shows a short code and verification URL
* User signs in on a phone or laptop
* Device polls until authorization is complete

=== Password Flow Caution

* Legacy fallback
* Application collects the user's password
* Weak fit for MFA, SSO, federation, and passwordless
* Prefer Authorization Code + PKCE as previously discussed.
