=== Cookies
image::iStock-1075835580.jpg[background, size=cover]

=== Two Sessions, Two Cookies

* Keycloak keeps its own Single Sign-On session
* Web Application keeps a separate local session
* Browser carries cookies for both systems
* Cookies are sent automatically on matching requests
* Automatic delivery is useful and security-sensitive

=== Essential Cookie Settings

[cols="30b,70"]
|===
| Setting | Why It Matters

| `HttpOnly`
| Browser JavaScript cannot read the cookie

| `Secure`
| Browser sends the cookie only over HTTPS

| `SameSite`
| Controls when the cookie is sent on cross-site requests

| `Path`
| Limits which Web Application paths receive it

| `Max-Age` / `Expires`
| Controls how long the Browser keeps it
|===

=== SameSite and OIDC

[cols="25b,75"]
|===
| Value | Behavior

| `Strict`
| Strong cross-site restriction; can break login callbacks

| `Lax`
| Often works for standard top-level login redirects

| `None`
| Allows cross-site use; requires `Secure`
|===

NOTE: Test cookie behavior using the same domains, HTTPS setup, and Browser behavior as production.

=== Keep Each Secret in Its Place

[cols="35b,65"]
|===
| Location | Keep Here

| Keycloak
| Private signing keys, user credentials, SSO session

| Web Application
| Local session state, Refresh Tokens, client credentials

| Browser
| Hardened session cookie; avoid long-lived tokens in JavaScript

| Backend Service
| Accepted issuers, audiences, and local permission decisions
|===

=== CSRF
image::iStock-1075835580.jpg[background, size=cover]

=== Cross-Site Request Forgery

* Malicious site causes an unwanted request
* Browser automatically attaches the Web Application cookie
* Attacker does not need to read the response
* State-changing requests create the risk

[source,html]
----
<form method="post" action="/profile/email">
  <input type="hidden" name="_csrf" value="a9f31c...">
  <input type="email" name="email">
</form>
----

=== CSRF Token vs OIDC State

[cols="35b,65"]
|===
| Value | Protects

| CSRF Token
| State-changing requests to the Web Application

| OIDC `state`
| Correlation between login request and callback
|===

* Generate unpredictable values
* Validate before accepting the request
* One does not replace the other

=== Logout Is a Real Example

* `GET /logout` is easy to trigger from another site
* Use a state-changing request with CSRF protection
* Local logout clears the Web Application session
* OIDC logout also clears the Keycloak SSO session

=== CORS and Web Origins
image::iStock-1075835580.jpg[background, size=cover]

=== CORS and Web Origins

* CORS controls which browser origins may read cross-origin responses
* Keycloak Web Origins configures the Browser policy for a Web Application
* Redirect URIs control where Keycloak may return login responses
* Web Origins and Redirect URIs solve different problems
* Avoid wildcard origins in production

WARNING: CORS does not stop curl, Backend Services, or malicious non-browser callers. It is not authorization.

=== Reverse Proxy Checklist

* `Secure` cookies require HTTPS
* Proxy preserves the original request scheme and host
* Web Application trusts forwarding headers only from known proxies
* Generated Redirect URIs match the public URL
* Production cookie behavior is tested end to end
