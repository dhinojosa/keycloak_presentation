[.chapter]
== Browser Security
image::iStock-1181695869.jpg[background, size=cover]

=== Why This Matters for OIDC

* OIDC browser login is built on redirects
* Redirects carry authorization codes through the browser
* The application usually keeps a local session after login
* That local session is usually represented by a cookie
* CSRF and cookie settings decide whether that session is safe

[.chapter]
=== CSRF
image::iStock-1075835580.jpg[background, size=cover]

=== Cross-Site Request Forgery

* CSRF tricks a logged-in browser into submitting a request
* The attacker does not need to read the response
* The browser automatically includes matching cookies
* The danger is state-changing requests:
** logout
** profile updates
** password changes
** purchases
** admin actions

=== CSRF Defense

* Require unsafe requests to include a CSRF token
* The token is generated by the application
* The token is not automatically added by another site
* `GET` should be safe and idempotent
* Use `POST`, `PUT`, `PATCH`, or `DELETE` for state changes

=== OIDC and CSRF

* OIDC uses `state` to protect the login redirect
* The application sends `state` to Keycloak
* Keycloak sends the same `state` back with the authorization code
* The application rejects the callback if `state` does not match
* This protects the login flow, not every application form

=== OIDC Callback Safety

* Validate `state`
* Use PKCE
* Lock down Redirect URIs
* Use HTTPS in production
* Reject unexpected issuer or audience

=== Logout Is a Real Example

* A logout link using `GET /logout` is easy to trigger cross-site
* Spring Security expects `POST /logout` when CSRF is enabled
* The logout form needs a CSRF token
* Local logout clears the application session
* OIDC logout should also clear the Keycloak SSO session

[.chapter]
=== Cookies
image::iStock-1075835580.jpg[background, size=cover]

=== What Cookies Do Here

* Keycloak uses cookies for its SSO session
* Spring and Quarkus use cookies for their application sessions
* Cookies are scoped by domain, path, scheme, and attributes
* Cookies are sent automatically by the browser
* That automatic behavior is powerful and risky

=== Essential Cookie Settings

[cols="1,2", options="header"]
|===
| Setting | Why it matters

| `HttpOnly`
| JavaScript cannot read the cookie

| `Secure`
| Browser only sends the cookie over HTTPS

| `SameSite`
| Controls when cookies are sent on cross-site requests

| `Path`
| Limits where the cookie is sent

| `Max-Age` / `Expires`
| Controls persistence
|===

=== SameSite

* `Strict`
** Best cross-site protection
** Can break normal login redirects
* `Lax`
** Good default for many session cookies
** Allows top-level navigations back to the application
* `None`
** Required for some cross-site iframe or third-party cases
** Must also use `Secure`

=== SameSite and OIDC

* OIDC needs the browser to leave the application and return
* `SameSite=Strict` can be too strict for login callbacks
* `SameSite=Lax` often works for standard top-level redirects
* `SameSite=None; Secure` is common when identity crosses site boundaries
* Test this in the same browser behavior your users will use

=== CORS and Web Origins

* CORS controls which browser origins can call APIs
* Keycloak clients can restrict allowed web origins
* Redirect URIs control where login responses can go
* Web origins control where browser requests can come from
* Do not use wildcards in production

=== Development vs Production

* Local demos often use plain HTTP
* Production should use HTTPS
* `Secure` cookies require HTTPS
* Reverse proxies must preserve the original scheme
* Misconfigured proxy headers can break redirect URIs and cookie behavior
