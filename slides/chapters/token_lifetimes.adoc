== Token Lifetimes & Refresh Tokens
image::iStock-1181695869.jpg[background, size=cover]

=== Token Lifetimes

* Access Tokens are intentionally short-lived
* Limits the impact of a stolen token
* Reduces the need for immediate token revocation
* Encourages periodic revalidation of identity
* Improves the overall security of distributed systems

=== Two Tokens: Two Responsibilities

* Access Token is presented to applications and APIs
* Refresh Token is presented only to Keycloak
* Access Token is short-lived (minutes)
* Refresh Token is longer-lived (hours or days)
* Each token has a different security purpose

=== Refresh Token Flow

* User authenticates with Keycloak
* Keycloak issues an Access Token and Refresh Token
* Applications use the Access Token until it expires
* Refresh Token requests a new Access Token
* User logs in again only after the Refresh Token expires or is revoked

=== Refresh Token Rotation

* Keycloak can issue a new Refresh Token during refresh
* The old Refresh Token should no longer be reused
* Reuse may indicate theft or a broken client
* Rotation limits the value of a stolen Refresh Token
* Tune this per client and application risk

=== Hotel Room Key Analogy

* Access Token is your hotel room key
* Refresh Token is your hotel reservation
* A lost room key has limited value because it expires
* The reservation stays at the front desk, not in the room
* Keycloak issues a new room key without requiring a new login

=== Token Storage

[cols="35b,65"]
|===
| Token / State | Recommendation

| Application session
| Prefer server-side session with hardened cookies: `HttpOnly`, `Secure`, `SameSite` (covered next)

| Access Token
| Keep short-lived; do not treat it like a password

| Refresh Token
| Protect aggressively; avoid exposing it to browser JavaScript

| Native / CLI / device
| Use platform storage; the threat model differs from browser login
|===

WARNING: Browser JavaScript is a weak place for long-lived credentials, especially Refresh Tokens.

=== Keep Each Secret in Its Place

[cols="35b,65"]
|===
| Boundary | Must protect

| Keycloak
| Private signing keys, user credentials, SSO sessions

| Application
| Refresh Tokens, session state, client credentials

| API
| Authorization decisions, accepted audiences, trusted issuers

| Browser
| Session cookie; avoid long-lived credentials in JavaScript

| Operations
| TLS keys, database credentials, backup access, admin accounts
|===
