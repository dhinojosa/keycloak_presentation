== Lab 2: Quarkus with Keycloak and OIDC

. Keep Keycloak and Postgres running from the previous lab.
+
[source,bash]
----
docker compose ps
----
+
. If Keycloak is not running, start it from the project root.
+
[source,bash]
----
docker compose up -d
----
+
. Open the OIDC Discovery Document again.
+
[source,bash]
----
curl http://localhost:8080/realms/keycloak-demo/.well-known/openid-configuration
----
+
. Find the following URLs in the Discovery Document.
.. `authorization_endpoint`
.. `token_endpoint`
.. `jwks_uri`
.. `issuer`
. Open the Quarkus configuration in `projects/quarkus-app/src/main/resources/application.properties`.
+
[source,properties]
----
quarkus.oidc.application-type=web-app
quarkus.oidc.auth-server-url=${QUARKUS_OIDC_AUTH_SERVER_URL:http://127.0.0.1:8080/realms/keycloak-demo}
quarkus.oidc.connection-delay=2S
quarkus.oidc.connection-retry-count=1
quarkus.oidc.connection-timeout=2S
quarkus.oidc.client-id=quarkus-app
quarkus.oidc.credentials.secret=${QUARKUS_OIDC_CREDENTIALS_SECRET:quarkus-secret}
quarkus.oidc.authentication.redirect-path=/oidc/callback
quarkus.oidc.authentication.restore-path-after-redirect=true
quarkus.oidc.roles.role-claim-path=realm_access/roles
----
+
. Notice what Quarkus needs from Keycloak.
.. `application-type=web-app` tells Quarkus to use browser redirects
.. `auth-server-url` points at the realm
.. The connection settings make demo failures surface quickly instead of waiting through long retries
.. `client-id` and `credentials.secret` identify the app
.. `redirect-path` is where Quarkus receives the authorization code
.. `restore-path-after-redirect` sends the user back to the original protected page after the callback completes
.. `roles.role-claim-path` tells Quarkus where Keycloak stores realm roles
. Open the Quarkus resource in `projects/quarkus-app/src/main/java/com/evolutionnext/keycloakdemo/quarkus/PageResource.java`.
.. Find `@RolesAllowed("user")`
.. Find `@RolesAllowed("admin")`
.. Find `@RolesAllowed("engineering")`
.. Find `@RolesAllowed("sales")`
. Start the Quarkus app from the `projects/quarkus-app` directory.
+
[source,bash]
----
mvn quarkus:dev
----
+
.. The project includes `.mvn/jvm.config` with `--add-opens=java.base/java.lang=ALL-UNNAMED`
.. The Quarkus Maven plugin also sets `<jvmArgs>--add-opens=java.base/java.lang=ALL-UNNAMED</jvmArgs>`
.. This avoids a JBoss Threads access error when running Quarkus dev mode on Java 24 or newer
. Open the public Quarkus page.
.. Go to `http://localhost:8082`
.. This page should load without a login
. Open the user page.
.. Go to `http://localhost:8082/user`
.. Quarkus redirects to Keycloak
.. Keycloak authenticates the user
.. Keycloak redirects back to Quarkus at `/oidc/callback`
.. Sign in as `user` with password `password`
.. The user page should show the principal and roles
. Try a page that the `user` account cannot access.
.. Go to `http://localhost:8082/admin`
.. The request should be denied because `user` does not have the `admin` role
. Log out or use a private browser window and try the other role pages.
.. Use the logout button on the protected page
.. Notice Quarkus handles `/logout`
.. Quarkus redirects through Keycloak's end-session endpoint
.. Use `admin` / `password` for `http://localhost:8082/admin`
.. Log out again
.. Use `engineer` / `password` for `http://localhost:8082/engineering`
.. Log out again
.. Use `sales` / `password` for `http://localhost:8082/sales`
. Inspect the browser cookies for `http://localhost:8082`.
.. Look for the Quarkus OIDC session cookie
.. The browser keeps a session with Quarkus
.. Quarkus uses the authorization code to obtain tokens from Keycloak
.. Quarkus validates token information using the realm metadata and keys
.. After logout, confirm the Quarkus session cookie is removed or replaced
. Open the Keycloak client configuration.
.. In the Admin Console, go to *Clients*
.. Open `quarkus-app`
.. Look at *Valid redirect URIs*
.. Confirm `http://localhost:8082/oidc/callback`
.. Look for the `realm roles` mapper
.. Confirm it maps roles into `realm_access.roles`
. Compare the Spring and Quarkus OIDC setup.
.. Both apps use the same realm
.. Both apps use a client id and client secret
.. Both apps redirect to Keycloak for login
.. Both apps receive an authorization code on a callback URL
.. Both apps use Keycloak roles to authorize pages
. After completing the lab, discuss the following questions.
.. Which Quarkus property points to the realm?
.. Which path receives the authorization code?
.. How does Quarkus find the user's roles?
.. What is the same between Spring and Quarkus?
.. What is different?
.. Why is app-only logout different from OIDC logout?
