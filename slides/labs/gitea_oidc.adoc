== Lab 3: Integration with a 3rd Party Application

. Start the full demo stack from the project root.
+
[source,bash]
----
docker compose up -d
----
+
. Confirm the containers are running.
+
[source,bash]
----
docker compose ps
----
+
. Open Gitea.
.. Go to `http://localhost:3000`
.. Gitea is the third-party application in this lab
.. Keycloak is still the Identity Provider
.. The same `keycloak-demo` realm is used
. Inspect how Gitea is started in `docker-compose.yml`.
+
[source,yaml]
----
gitea:
  image: gitea/gitea:1.22.6
  command:
    - /bin/sh
    - -c
    - |
      /etc/s6/gitea/setup
      cat >/tmp/keycloak-forward.sh <<'EOF'
      #!/bin/sh
      exec nc keycloak 8080
      EOF
      chmod +x /tmp/keycloak-forward.sh
      nc -lk -p 8080 -e /tmp/keycloak-forward.sh &
      su-exec git gitea migrate
      until wget -q -O /dev/null http://localhost:8080/realms/keycloak-demo/.well-known/openid-configuration; do
        echo "Waiting for Keycloak OIDC discovery..."
        sleep 2
      done
      AUTH_ID="$$(su-exec git gitea admin auth list | awk '/keycloak/ {print $$1; exit}')"
      if [ -n "$$AUTH_ID" ]; then
        su-exec git gitea admin auth update-oauth \
          --id "$$AUTH_ID" \
          --name keycloak \
          --provider openidConnect \
          --key gitea \
          --secret gitea-secret \
          --auto-discover-url http://localhost:8080/realms/keycloak-demo/.well-known/openid-configuration \
          --scopes openid \
          --scopes profile \
          --scopes email
      else
        su-exec git gitea admin auth add-oauth \
          --name keycloak \
          --provider openidConnect \
          --key gitea \
          --secret gitea-secret \
          --auto-discover-url http://localhost:8080/realms/keycloak-demo/.well-known/openid-configuration \
          --scopes openid \
          --scopes profile \
          --scopes email
      fi
      exec su-exec git gitea web
----
+
. Notice the important parts of the automatic Gitea connection.
.. The small `nc` process forwards `localhost:8080` inside the Gitea container to the `keycloak:8080` service
.. This keeps the OIDC URLs browser-friendly as `http://localhost:8080/...`
.. `gitea migrate` initializes the Gitea database
.. The `wget` loop waits until Keycloak's Discovery Document is available
.. `gitea admin auth list` checks whether the OIDC source already exists
.. `gitea admin auth update-oauth` repairs an existing `keycloak` source
.. `gitea admin auth add-oauth` adds Keycloak as an OpenID Connect login source when missing
.. `--name keycloak` becomes the provider name in the Gitea login URL
.. `--provider openidConnect` tells Gitea to use OIDC
.. `--key gitea` is the Keycloak client id
.. `--secret gitea-secret` is the Keycloak client secret
.. `--auto-discover-url` points Gitea at the Keycloak Discovery Document
. Confirm that Gitea has the Keycloak auth source.
+
[source,bash]
----
docker compose exec gitea gitea admin auth list
----
+
. Open the OIDC Discovery Document that Gitea uses.
+
[source,bash]
----
curl http://localhost:8080/realms/keycloak-demo/.well-known/openid-configuration
----
+
. Find the endpoints Gitea needs.
.. `authorization_endpoint`
.. `token_endpoint`
.. `userinfo_endpoint`
.. `jwks_uri`
.. `issuer`
. Open the Keycloak Admin Console.
.. Go to `http://localhost:8080/admin`
.. Sign in with username `admin` and password `admin`
.. Select the `keycloak-demo` realm
. Open the Gitea client in Keycloak.
.. Go to *Clients*
.. Open `gitea`
.. Confirm the client id is `gitea`
.. Confirm the client is confidential
.. Confirm the valid redirect URI is `http://localhost:3000/user/oauth2/keycloak/callback`
. Log in to Gitea with Keycloak.
.. Open `http://localhost:3000`
.. Choose *Sign In*
.. Choose the `keycloak` OpenID Connect provider
.. Sign in as `user` with password `password`
.. Keycloak authenticates the user
.. Keycloak redirects back to Gitea at `/user/oauth2/keycloak/callback`
.. Gitea creates or links a Gitea account for the Keycloak identity
. Compare this with the Spring and Quarkus labs.
.. Spring used framework properties to configure OIDC
.. Quarkus used framework properties to configure OIDC
.. Gitea uses an application-level OAuth/OIDC login source
.. All three rely on the same Keycloak realm
.. All three rely on the same OIDC Discovery Document pattern
. After completing the lab, discuss the following questions.
.. Which Keycloak realm did Gitea use?
.. What is Gitea's OIDC client id?
.. What URL does Gitea use as its redirect URI?
.. Why does Gitea need the Discovery Document?
.. What is different between integrating a framework app and integrating a third-party product?
